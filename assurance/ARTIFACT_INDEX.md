## Public sample integration

The API and console load strict `AssessmentPackage` records from:

- `data/sample-runs/baseline/package.json`
- `data/sample-runs/remediated/package.json`

Each directory also contains source-oriented run, evidence, results, assessment, lifecycle, and diff records plus a verified `run-manifest.json`. Checked-in signatures are local CI/sample signatures and use a `local://` key ID; no private key is committed and no Key Vault provenance is claimed.

Both package records embed the strict system record loaded from `config/system-record.json`. Each manifest uses schema version `1.1.0` and carries a CAD cost breakdown for model, compute, storage, and telemetry with a validated component total.

Replay collectors load twelve source envelopes per state from `data/collector-fixtures/{baseline,remediated}`. Behavioral, mapping, integrity-bound policy, and scenario datasets are under their corresponding `data/` directories.

`data/scenario-campaigns/controlled-execution.json` is the executable, cross-linked lifecycle
proof for all eight `SCN-*` specifications. It binds clean/injected/retest results to actual
RuleEngine, OPA/Rego, behavioral-runtime, and signed-sample evidence as applicable, and records
remediation, risk, closure/reopen semantics, and cleanup. It intentionally contains no claim of
Azure-live execution; the release-required external reruns are identified per scenario.

`data/ai-evaluations/replay-results.json` is generated by running all fixed cases through `PolicyAssistantService` with the replay adapter. It contains versioned adapter, configuration, corpus, interaction, retrieval, guardrail, tool, and latency observations, plus response digests rather than response prose. Generation and the external Foundry/Phi gate are documented in `docs/operations/behavioral-evaluations.md`.

## Deployment-readiness handoffs

- `scripts/azure/prepare-corpus-handoff.py` creates and verifies the deterministic create-only
  corpus bundle and validates the protected Azure MCP Blob readback receipt.
- `infra/entra/app-registration-handoff.json` fixes the two Entra resource applications, human and
  managed-identity app roles, callback sources, audiences, and assignment expectations.
- `scripts/azure/prepare-entra-handoff.py` binds foundation callbacks into the MCP request and
  verifies the private application/service-principal/assignment readback receipt.
- `docs/operations/azure-deployment-handoffs.md` is the operator runbook. No checked-in artifact
  claims these external Azure gates have run.

## Validation

```bash
PYTHONPATH=src .venv/bin/python assurance/scripts/score_mapping_benchmark.py
PYTHONPATH=src .venv/bin/python assurance/scripts/validate_contracts.py
PYTHONPATH=src .venv/bin/python assurance/scripts/validate_oscal.py
PYTHONPATH=src .venv/bin/python assurance/scripts/validate_artifacts.py
PYTHONPATH=src .venv/bin/python assurance/scripts/validate_scenarios.py --conftest /tmp/conftest
```

`validate_oscal.py` performs strict validation against the bundled, checksum-pinned
official NIST OSCAL v1.2.2 complete JSON schema. The development dependencies
provide `jsonschema` and ECMA-262-compatible Unicode regular expressions. To
cross-check with a separately downloaded official bundle, set `OSCAL_SCHEMA_DIR`
or pass `--schema-dir`.
